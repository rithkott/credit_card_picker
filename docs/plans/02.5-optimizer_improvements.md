# Step 2.5: Optimizer scale improvements — design

Complete design for two exact (result-preserving) changes to
`scripts/optimize.py`, spec'd here first per the 02-optimizer.md precedent and
implemented in the same change set as this doc:

1. **Dynamic subset budget** — replace the static `MAX_ELIGIBLE_CARDS = 80`
   gate with a gate on the actual enumerated work, `Σ C(n,k), k = 1..max_cards`.
2. **Exact dominance pruning** — a per-run, pre-search pass that removes card
   variants provably unable to appear in an optimal portfolio.

Motivation: the 139-card dataset expands to **129 eligible variants** on the
example profile, past the old `n ≤ 80` hard stop, so every run currently exits
2. Both changes preserve the optimizer's core promise — deterministic,
provably optimal portfolios — rather than switching to a heuristic search.
This supersedes the "80 eligible cards" language in `02-optimizer.md` §6/§10;
those sections carry a pointer here. Beam search / branch-and-bound (the §6
"~200 cards" path) remains future work; this step keeps the search exhaustive.

> **Superseded in part:** branch-and-bound shipped as the default engine (with
> exhaustive kept verbatim as the oracle), and exact assignment closed this
> doc's §2.4 regret-rerouting caveat. Dominance pruning now defaults OFF under
> the bnb engine (ranked-rows fidelity restored) and stays ON for exhaustive.
> See `docs/plans/10-optimizer-overhaul.md`.

Scope notes:

- No schema, registry, or profile-format changes. A "skip student cards / low
  credit-tier cards" filter was considered and rejected: it needs a new schema
  field, can wrongly exclude a genuinely optimal card, and is subsumed by
  dominance pruning (a student card that is a strictly worse clone of its
  sibling is pruned automatically for any user eligible for both).
- Two Discover card files (`it-cash-back`, `it-student-cash-back`) had
  `rotating` rewards with no `cap`, which `build_lines` rejects by design —
  fixed in this change with the published $1,500/quarter cap (marked NEEDS
  human verification, `confidence: low` unchanged).
- The golden-test suite is repointed at a frozen fixture copy of the 8 seed
  cards (`tests/fixtures/data/`) so dataset growth never invalidates
  hand-computed goldens again (§5).

## 1. Dynamic subset budget

### 1.1 The measured constraint

`search()` scores every subset with a full joint `score_portfolio` pass,
measured at **~36 µs per subset at k=3, ~55 µs at k=5** (pure Python, Apple
Silicon, example profile). Subset counts `Σ C(129,k)`:

| max_cards | subsets     | est. runtime |
|-----------|-------------|--------------|
| 3         | 357,889     | ~13 s        |
| 4         | 11,367,265  | ~8 min       |
| 5         | 286,601,665 | ~4.4 h       |

The old `n ≤ 80` gate was calibrated to the worst case (`max_cards: 5`) and so
rejected profiles that would actually run in seconds. Gating on the real work
admits today's dataset at the default `max_cards: 3` and fails cleanly where
runtime would genuinely be unreasonable.

### 1.2 Policy constant

```python
# Exhaustive search scores every subset; each score_portfolio call costs
# ~35-55 µs in pure Python. 2M subsets ≈ one to two minutes — the tolerable
# ceiling for an interactive CLI. At max_cards=3 this admits ~229 variants
# (C(229,3) ≈ 2M), matching §6's ~200-card horizon.
MAX_SCORED_SUBSETS = 2_000_000
```

Replaces `MAX_ELIGIBLE_CARDS` (deleted, including its `policy_constants()`
echo — `MAX_SCORED_SUBSETS` is echoed instead; the echoed block changes bytes,
which is fine: determinism is identical inputs ⇒ identical output, not
stability across code versions).

### 1.3 The gate

```python
def subset_budget(n_variants: int, max_cards: int) -> int:
    """Upper bound on scored subsets: sum of C(n, k) for k = 1..max_cards.
    Ignores the same-base_id exclusion (a small documented over-count)."""
    return sum(math.comb(n_variants, k) for k in range(1, max_cards + 1))
```

In `search()`, after the effective `max_cards = min(user max, distinct
base_ids)` is computed (using the effective value is deliberately more
lenient), raise `DataError` when `subset_budget(len(ids), max_cards) >
MAX_SCORED_SUBSETS`, with a message stating the variant count, the subset
count, and the remedy (lower `user.max_cards`). The bound intentionally does
not subtract combinations excluded by the same-`base_id` rule: `Σ C(n,k)` is a
cheap, honest upper bound and the correction is <3% today.

## 2. Dominance pruning — placement and shape

```python
# in run(), between expansion and search:
eligible, excluded = filter_cards(dataset["cards"], profile)
variants = expand_choice_variants(eligible, profile)
variants, pruned = prune_dominated_variants(variants, profile, mode,
                                            programs, merchants)
ranked = search(variants, profile, mode, programs, merchants, as_of)
```

`prune_dominated_variants(variants, profile, mode, programs, merchants) ->
(kept, pruned)` is a pure function; `pruned` is an id-sorted list of
`{"id": ..., "reason": "dominated by <id>"}`. It is **profile-aware**: rates
come from `build_lines` under this profile's `valuation_mode` cpp,
`uses_travel_portal` (since plan 07: the per-portal `confirmed_usage` gate),
`activates_rotating`, and closed-loop filtering, and
only buckets with spend > `EPS` are binding. It is deliberately **not** called
inside `search()` — callers that hand-build variant lists (tests, future
tooling) keep exact search semantics. It takes no `as_of`: a card with an
expired signup bonus still blocks its own pruning (conservative, and keeps
the function time-free).

### 2.1 Per-variant rate tables

For each variant, from its `build_lines` output over live buckets
(`amount > EPS`):

- `best_any[b]` — max `effective_rate` over lines eligible for bucket `b`
  (capped lines count at their headline rate).
- `best_uncapped[b]` — max `effective_rate` over lines with `room is None`
  eligible for `b` (base and fallback lines qualify; capped lines never do —
  inside a subset their room may already be consumed).
- Card-level: `fee` (`annual_fee_usd`), `year1_fee` (0 when
  `first_year_waived`), `has_extras` (non-empty `credits` or non-null
  `signup_bonus`), `has_clamp` (`max_annual_rewards_usd` present), `base_id`.

The pass is O(n) table building + O(n² × buckets) pairwise checks — n ≈ 129,
~10 buckets, sub-millisecond.

### 2.2 The criterion

Variant **B dominates A** (A ≠ B) iff **all** of:

- **(b) A is plain** — `A["credits"] == []` and `A["signup_bonus"] is None`.
  Conservative: even an expired bonus or a credit the profile can't use blocks
  pruning. Cheap honesty; 39 of today's 129 variants are plain.
- **(a) uncapped pointwise cover** — for every live bucket `b` where A has any
  eligible line: `best_uncapped_B[b] ≥ best_any_A[b] − EPS`. Only B's
  *uncapped* rates count (its capped room may be spent in-subset); A is
  measured at its best headline rate. Closed-loop, `portal_only`, and rotating
  activation are respected automatically because both tables come from
  `build_lines` under this profile.
- **(c) fees** — `fee_B ≤ fee_A` and `year1_fee_B ≤ year1_fee_A`.
- **(d) no clamp on B** — B has no `max_annual_rewards_usd` (absorbing A's
  spend must not be able to hit a ceiling). A may be clamped.
- **(g) match-interception guard** — for every live bucket `b` and every
  eligible variant M ∉ {A, B} whose `signup_bonus` has `first_year_match`:
  **not** (`best_uncapped_A[b] < best_any_M[b] ≤ best_any_B[b]`), with EPS
  tolerance, ties counting as interception. In words: swapping A→B may not
  newly out-rate a match card anywhere A's uncapped rates didn't already.
  Applied regardless of `optimize_for` (it also protects the `year1_net`
  tie-break). See §2.4 for why this exists.
- **(s) sibling rule** — every variant sharing B's `base_id` (including B
  itself) must satisfy (a), (c), (d), (g) versus A. The swap argument inserts
  B into `S \ {A}`, but if S already contains a *different configuration* of
  B's physical card, B cannot be inserted; the sibling already present must
  itself cover A. Nearly free in practice: choice variants differ only in one
  capped category line, and capped lines never count for (a).
- **(e) strictness / lex tie rule** — if every binding dimension is exactly
  equal (both fees, and `best_uncapped_B[b] == best_any_A[b]` on every live
  A-bucket — no strict advantage anywhere), prune only when
  `B["id"] < A["id"]`. With any strict advantage, prune unconditionally. This
  keeps the reported portfolios deterministic under the ranking's
  lexicographic tie-break: among exact clones, the lexicographically smaller
  id survives, which is the one the tie-break would have reported anyway.

**Pruning is simultaneous**: A is pruned if *any* B dominates it, whether or
not B is itself pruned. The `reason` names the lexicographically smallest
dominator. This is safe because the relation is acyclic: strict domination is
transitive (an uncapped B-line on bucket `b` witnesses `b` in B's eligibility,
so a dominator of B must cover `b` at ≥ that rate) and asymmetric (mutual
strict domination would force `best_uncapped_B[b] ≥ best_any_A[b] >
best_any_B[b] ≥ best_uncapped_B[b]`, a contradiction), and tie edges are
ordered by id — so every pruned variant heads a finite chain ending at a
surviving transitive dominator.

### 2.3 Why this is exact (the swap argument)

Scoring is pool-independent: `score_portfolio` sees only the subset's cards,
so pruning removes rows from the ranked list without changing any surviving
subset's score. For any subset `S ∋ A`, map `S → S′ = S \ {A} ∪ {B}` (or
`S \ {A}` when B or a dominating sibling is already in S — subsets of size
k−1 ≥ 1 are enumerated too). Then:

- Every dollar A's lines could win is caught by an uncapped B line at ≥ rate
  (condition a), so total spend earnings are weakly ≥.
- A contributes no credits or bonus (condition b); B's extras only add. The
  shared per-category credit tracker means B's credits can shrink a later
  card's categorized credit by at most what B gains — totals weakly rise.
- Fees are weakly ≤ in both metrics (condition c); B cannot be clamped
  (condition d).

Hence `ongoing_net(S′) ≥ ongoing_net(S)` — within the same structural scope
where §5.5 of 02-optimizer.md declares the greedy assignment exact — so at
least one optimal portfolio survives pruning, and with condition (e) the
reported rank-1 portfolio itself is stable.

### 2.4 The `first_year_match` hole and the honesty note

`year1_net` is not purely additive: a `first_year_match` bonus equals the
card's *own* assigned earnings, so shifting spend between cards inside a
subset can lower `year1_net` even while total earnings rise. Two concrete
failure shapes the plain swap argument misses:

1. B's lines (capped ones included, on buckets A never touched) out-rate a
   match card M's line, stealing spend M held in S and halving its year-1
   contribution.
2. A's *capped* line out-rated M with limited room, and M caught the overflow
   in S; B's uncapped cover absorbs all of it in S′.

Condition (g) closes both: B may not out-rate any eligible match card on any
live bucket unless A's *uncapped* rate already did (using A's uncapped rate —
not `best_any` — is what closes shape 2). The guard is per-(A, B, bucket)
rate arithmetic; it does **not** globally disable pruning when match cards
(Discover it: rotating + match) are eligible — their 5% rotating headline
sits above realistic flat dominators, and base-rate ties count as already
intercepted, so the flat-card prunings that matter pass.

**Honesty note** (precedent: 02-optimizer.md §5.5): one residual third-order
path remains — inserting B can reshape a *third* card's rotating regret
routing (best-alternative rates change), relocating which buckets its capped
room consumes and thereby shifting a match card's leftovers. Guarding against
it would require simulating full assignments per candidate pair, defeating the
purpose of a cheap pre-pass. The guarantee is therefore: **exact for
`ongoing_net` within §5.5's greedy-exactness scope; exact for `year1_net`
against direct interception (guard g); indirect regret-rule rerouting through
a third card is accepted as within the greedy's documented heuristic scope.**
Pruning also intentionally removes dominated-card portfolios from the ranked
list entirely — the guarantee is that at least one optimal portfolio survives,
not that sub-optimal rows are preserved.

## 3. Output contract changes

Never silently drop (02-optimizer.md §7): pruned variants are surfaced
everywhere excluded cards are.

- `run()` bundle: `card_variants` keeps its existing meaning (post-expansion,
  **pre-prune** count); add `"card_variants_pruned": len(pruned)` and
  `"pruned": pruned` (id-sorted `{"id", "reason"}` dicts).
- `render_text` header: the Cards line becomes
  `"... ({V} variants after choose-your-own-category expansion, {P} pruned as
  dominated), {E} excluded (...)"`, and when P > 0 a
  `"Pruned: id1 (dominated by x); id2 (dominated by y)"` line follows
  (id-sorted). Both text and JSON stay byte-deterministic.
- `policy_constants()` echoes `MAX_SCORED_SUBSETS` (replacing
  `MAX_ELIGIBLE_CARDS`).

## 5. Test plan (`tests/test_optimizer.py`)

**Fixture freeze.** The suite currently loads the live dataset while pinning
hand-computed values from the original 8 seed cards — 23/51 tests fail after
the 139-card import, before this change. Fix: freeze the seed dataset (8 card
YAMLs + `categories.yaml`, `merchants.yaml`, `point-valuations.yaml`) under
`tests/fixtures/data/`, taken from the last git revision where the suite was
green, and point the test module at it by assigning `opt.CARDS_DIR` /
`opt.META_DIR` before `DATASET = opt.load_dataset()` (module-level; this also
routes `TestDeterminism`'s `opt.main()` through the fixture). All 51 existing
tests must pass unchanged — including `test_search_is_exhaustive_and_ranked`'s
`len(results) == 85`, which calls `search()` directly and is untouched by
`run()`-level pruning (and all 8 seed cards have credits or a bonus, so
condition (b) would exempt every one regardless).

**TestSubsetBudget (new).** `subset_budget(129,3) == 357_889` and
`subset_budget(129,5) == 286_601_665`; gate raises `DataError` before any
scoring on an over-budget synthetic pool (message names `max_cards` and
`MAX_SCORED_SUBSETS`); small pools search fine; `policy_constants()` contains
`MAX_SCORED_SUBSETS` and not `MAX_ELIGIBLE_CARDS`.

**TestDominancePruning (new, `synth_card`-based).**

- Strictly-worse clone (1% base vs 2% base, both $0-fee, no extras) pruned
  with reason `"dominated by <id>"`.
- NOT pruned when: A has a `signup_bonus`; A has a credit; the dominator's
  covering line is capped (B = 5%-capped groceries + 1% base vs A = 1.5%
  uncapped groceries); B's fee higher, or A has `first_year_waived` and B
  doesn't; B carries `max_annual_rewards_usd`.
- Cap-room coexistence: A = 5%-capped-groceries no-extras card survives
  against B = 2% uncapped flat (A's headline beats B's uncapped cover).
- Match guard: with synthetic M (`first_year_match`, 3% dining) in the pool,
  B = 4% dining + 2% base does NOT prune A = 1% flat; control B = 2% flat
  does.
- Tie determinism: identical clones with ids `aaa`/`bbb` → `bbb` pruned under
  both input orders; the `pruned` list is byte-stable.
- `run()`-level integration on a small synthetic dataset: bundle carries
  `card_variants_pruned`/`pruned`; `render_text` shows the Pruned line.

**End-to-end** (manual, plus the determinism test): `python3
scripts/optimize.py --profile examples/spend-profile.example.yaml` completes
in well under a minute, header shows eligible/variants/pruned/excluded
counts, two runs are byte-identical; a `max_cards: 5` profile fails cleanly
with the budget message. Exactness spot-check: on a reduced pool small enough
for both, search with and without pruning agrees on the rank-1 portfolio and
value.
