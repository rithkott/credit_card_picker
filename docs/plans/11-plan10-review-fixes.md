## Root causes and fix plan

### R1 (CRITICAL) — bnb engine scores subsets in visitation order, not sorted order

`scripts/search_bnb.py:135` — `exact()` passes cards to `score_portfolio` in DFS/seeding
visitation order; the exhaustive engine scores every subset in sorted-id order.
`score_portfolio` is card-order-sensitive whenever the min-cost-flow assignment is adopted:
`assign_exact.solve_assignment` breaks rate ties by line-list position, so residual spend
attribution shifts between cards, which feeds `first_year_match` bonuses and
`max_annual_rewards_usd` clamps. The same subset therefore gets a different
`year1_net`/`ongoing_net` inside bnb's comparator than exhaustive computes.

Verified twice, independently, with end-to-end repros:
- Synthetic (shared cap pool + match card): bnb returns best 3-card portfolio worth $148 while exhaustive correctly finds $187 — a strictly suboptimal answer, and corrupt incumbents also poison pruning thresholds.
- Real dataset ({customized-cash, it-cash-back, it-miles, premium-rewards, reflect}): `it-miles` match bonus flips $147.18 vs $0 by order; bnb drops the true #3 portfolio.

Violates invariants (2) byte-identical bundles and effectively the exactness guarantee. bnb is the default engine.

**Fix:** canonicalize order inside `exact()` before scoring — `cards = [by_id[c] for c in sorted(card_ids)]`. Monkeypatch experiment already confirmed this restores byte-identity.

**Test gap to close:** `tests/fixtures/data/cards` contains no `first_year_match` or
`max_annual_rewards_usd` card, and `test_search_bnb.test_random_synthetic_pools` never
combines flow-solver adoption with order-sensitive attribution. Add a fixture card with
`first_year_match` + a shared-cap pool that forces flow adoption, and pin an
exhaustive-vs-bnb byte-equality test over it.

### R2 (MAJOR) — DB loader loses curated registry order

`scripts/optimize.py:204` — `load_dataset_db` rebuilds categories/merchants/programs with
`SELECT ... ORDER BY key`, while `load_dataset` preserves curated YAML file order;
`scripts/build_db.py` gives those three tables no position column (contrast: usage_groups /
usage_items store position and roundtrip correctly). Violates invariant (5); order is
observable through `GET /api/config`, which the site consumes for display ordering.

**Fix:** add a `position` column to categories/merchants/programs in `build_db.py`, insert in
YAML order, load with `ORDER BY position`. Bump artifact version/manifest so stale DBs
rebuild. Extend the roundtrip test to assert key order equality, not just content equality.

### R3 (MAJOR) — `load_state` unserialized; concurrent rebuilds corrupt the artifact

`server/app.py:92` — no lock: two threadpool requests can both fail the `ensure_fresh`
signature check and both enter `load_state`. `build_db.build()` writes in place after
unlinking, so one thread deletes the other's in-progress DB, and a reader can observe the DB
after DDL but before the insert transaction commits (empty tables, no meta row).

**Fix:** module-level `threading.Lock` around the `ensure_fresh` check-and-reload; re-check the
stat signature after acquiring the lock (double-checked locking) so only one thread rebuilds.
Also make `build()` write to a temp path and `os.replace()` into place so readers never see a
half-built artifact.

### R4 (MAJOR + minor dup) — stat signature captured after the load it guards

`server/app.py:121` — `STATE["stat_signature"]` is recorded *after* all sources are read. A
source file written during the ~1s load window is absorbed into the recorded signature while
the in-memory dataset, dataset_hash, and result cache still reflect the old content — the
change is permanently masked and stale results are served until the next unrelated edit.

**Fix:** capture the signature *before* reading any source; optionally re-stat after load and
retry once if it moved. Combine with the R3 lock in one patch — same function, same tests.

### U1 (minor, unverified) — flow-LP adoption ignores post-clamp value

`scripts/optimize.py:734` — `assign_spend` adopts the flow assignment on pre-clamp value, but
`max_annual_rewards_usd` clamps afterwards in `score_portfolio`, so the adopted assignment can
score below plain greedy post-clamp (bends invariant 4 at bundle level). Not adversarially
verified (filtered from the verify queue as minor). Verify first; if real, compare post-clamp
values when any card in the subset carries `max_annual_rewards_usd`.

### [minor] contract-and-tests — `server/app.py:121`

**Finding:** load_state records the hot-reload stat signature only after rebuilding the SQLite artifact and loading the dataset, so a source-file write that lands during the reload is absorbed into the recorded signature while the served dataset (and the manifest hash stamped into the artifact, which build() computes before re-reading the YAML for inserts) predates it — ensure_fresh then sees a matching signature on every subsequent request and never reloads, serving the stale dataset indefinitely until some other file changes.

**Failure scenario:** A card YAML is saved twice in quick succession (format-on-save editor, or git checkout switching branches) while a request is in flight: the request's ensure_fresh triggers load_state, which builds/loads from the first write's content, then _stat_signature() captures the second write's mtime_ns. All later requests compare equal signatures, skip reload, and /api/optimize keeps computing (and LRU-caching under the mislabeled dataset_hash) results from the outdated dataset until the next unrelated source edit. Capturing the signature before reading the sources closes the window.

**Verdict (CONFIRMED):** CONFIRMED by tracing server/app.py and by a live demonstration against a temp copy of the dataset. load_state (server/app.py:84-121) reads/builds from the YAML sources first (build_db.is_fresh + build_db.build both read every source file, then load_dataset_db, then two meta YAML reads) and only records STATE["stat_signature"] = _stat_signature() as its last step (line 121). Any source write landing inside that multi-hundred-ms window (build re-parses ~120 YAML files) is absorbed into the recorded signature while the served dataset predates it. ensure_fresh (line 124-129) is the ONLY automatic reload trigger and compares stat signatures for equality, so every subsequent request matches and never reloads. Demonstrated with the repo's real code paths on a temp dataset copy: write A triggers a reload; a write B injected between load_dataset_db and the signature capture leaves the server serving A's content (last_verified 2088-01-01) while disk says 2099-12-31; three further ensure_fresh calls stay stale and _stat_signature() == recorded signature; additionally STATE["dataset_hash"] no longer matches the actual disk manifest hash, so LRU results are cached under a mislabeled hash — a demonstrated violation of invariant (7) (stale serve across a dataset change). Refutation attempts failed: no lock, no secondary freshness check (is_fresh only runs inside load_state), mtime_ns granularity is irrelevant since the signature is captured after write B, POST /api/reload is manual only, and tests/test_server_api.py TestHotReload only covers writes landing between requests, never during load_state. The proposed fix works: simulating a signature captured before reading the sources makes the very next request detect the mismatch and reload to the on-disk content (verified in the same demo). Severity is genuinely minor — dev-only hot-reload path, needs a request in flight during a double-write burst (format-on-save, git checkout), and it self-heals on any later source edit or manual /api/reload — but the stale-serve-indefinitely failure mode is real and reproducible. Fix: capture sig = _stat_signature() at the top of load_state (before is_fresh/build/load) and store that value at the end; worst case is one redundant reload instead of an indefinite stale serve.

### [minor, unverified] assignment-exactness — `scripts/optimize.py:734`

**Finding:** assign_spend adopts the flow-LP assignment based on pre-clamp value, but score_portfolio applies max_annual_rewards_usd afterwards, so the adopted assignment can yield a lower post-clamp portfolio score than plain greedy (violates the never-worse-than-greedy intent of invariant 4 at the bundle level).

**Failure scenario:** Subset contains a card with max_annual_rewards_usd (dataset: sams-club-mastercard, $5,000 Sam's Cash cap) plus a detector-False cap competition. Verified numerically with the module's own line shapes: lines W=3.0 on {A,B} room 1000, C=2.9 on {B} room 1000, D=2.8 uncapped on {A}, base 1.0; buckets A=10000, B=2000. Greedy per-card earnings: D=25200, raw total 32100. The LP reroutes W's room to B and gives D all of A: D=28000, raw total 33900, so it is adopted (beats greedy by 1800 > VALUE_EPS). With D's reward clamp at 25200, score_portfolio (optimize.py:1129-1135) truncates D's earnings: adopted portfolio scores 31100 vs 32100 for greedy — a $1,000 regression versus the pre-plan-10 greedy engine for the same subset. In the real dataset this needs ~$167k+ of sams-eligible spend so the $5,000 clamp binds, hence rare, but the profile schema does not forbid it and both engines silently rank that subset too low. Fix direction: when a subset card carries max_annual_rewards_usd, compare post-clamp totals (or skip adoption) instead of raw assignment values.
