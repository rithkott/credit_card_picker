# Step 3: Manual Spend-Entry Form — design

Complete design for the manual category-spend entry form. This document is the
implementation spec for `tools/spend-profile-form.html`: a later session builds
exactly what is written here. Nothing in this step ships code, so
`docs/architecture.md` is untouched (it documents only what is built); the
implementation session must extend the diagram and the CLAUDE.md sync rule when
the form lands.

The form's one job is to produce a spend-profile YAML that
`parse_profile` in `scripts/optimize.py` accepts verbatim. That function is the
entire output contract — no schema fields, registry entries, or optimizer
changes are needed, and none are permitted as part of this step.

Delivery model is the same as `tools/card-entry-form.html`: a single-file,
no-build, vanilla HTML/CSS/JS page opened via `file://`. Framework and hosting
decisions are deferred to step 4 (product UI); to survive a later port, all
logic — state, validation, YAML emission — must be plain functions with no
DOM-framework coupling.

## 1. Scope and non-goals

The usage loop the form serves:

```
open tools/spend-profile-form.html → fill in estimated spend + "about you"
  → Download spend-profile.yaml
  → python3 scripts/optimize.py --profile spend-profile.yaml
```

**Non-goals for v1** (listed so nobody accidentally half-builds them):

- No Plaid, bank-linking, or CSV/statement import — manual estimates are the
  product decision (`docs/research.md`); precision theater is explicitly out.
- No monetization, affiliate links, or card promotion of any kind, ever.
- No persistence: no `localStorage`, no accounts. The downloaded YAML file *is*
  the persistence; re-running the optimizer with an edited file is the
  re-entry path.
- No YAML import / re-edit of an existing profile in v1 (deferred — see §7.3).
- No framework, no build step, no network fetches (the page must work from
  `file://`, where `fetch()` of local files is blocked anyway).
- No in-browser optimizer. The form emits input; `scripts/optimize.py` stays
  the only computation engine.
- No spend seasonality — annual (or monthly ×12) estimates only, matching the
  optimizer's even-spread assumption (`02-optimizer.md` §1).

## 2. Output contract

The emitted file must be accepted by `parse_profile`
(`scripts/optimize.py:153–219`) and match the documented example
(`examples/spend-profile.example.yaml`) in shape and commenting style. A full
sample emission, used as the golden acceptance case in §8:

```yaml
# Spend profile generated by tools/spend-profile-form.html.
# Run:
#   python3 scripts/optimize.py --profile spend-profile.yaml
# Full input rules: docs/plans/02-optimizer.md §2.

spend:                    # annual USD per category key from data/meta/categories.yaml
  groceries: 8000
  dining: 5000
  gas: 2000
  travel_flights: 1500
  travel_hotels: 1000
  streaming: 600
  other: 12000

merchant_spend:           # optional carve-outs, keys from data/meta/merchants.yaml;
  costco: 3000            # counted INSIDE spend[groceries], not in addition to it

user:
  credit_tier: very_good  # required: building | fair | good | very_good | excellent
  valuation_mode: floor   # floor | optimistic (default floor)
  max_cards: 3            # 1-5 (default 3)
  optimize_for: ongoing   # ongoing | year1 (default ongoing)
  activates_rotating: true
  uses_travel_portal: false   # SUPERSEDED (plan 07): now confirmed_usage — see
                              # docs/plans/07-confirmed-usage.md; the form must
                              # emit the questionnaire's confirmed keys instead
```

The rules, each traceable to a line in `parse_profile`:

1. Top-level keys are exactly a subset of `{spend, merchant_spend, user}`;
   anything else is an error (`optimize.py:158–160`).
2. `spend` is a required, non-empty mapping (`:162–164`). Every key must be a
   **non-pseudo** key from `data/meta/categories.yaml` (`:166–170` — `rotating`
   and `choice` are rejected), every value a non-boolean number ≥ 0 (`:171`,
   `_require_number`).
3. `merchant_spend` is optional; when present, every key must exist in
   `data/meta/merchants.yaml` (`:179–180`), every value a number ≥ 0. Per
   parent category, the sum of its merchants' carve-outs must not exceed
   `spend.get(category, 0)` (`:184–189`) — carve-outs are sub-buckets of their
   category, never additive. Note the `.get(cat, 0)`: a carve-out whose parent
   category is absent (or zero) is an error unless the carve-out is 0.
4. `user` is a required mapping; `credit_tier` is required and must be one of
   `building | fair | good | very_good | excellent` (`TIER_ORDER`,
   `optimize.py:68`). The five optional keys default exactly to `USER_DEFAULTS`
   (`optimize.py:76–78`): `valuation_mode: floor`, `max_cards: 3`,
   `optimize_for: ongoing`, `activates_rotating: true`,
   `uses_travel_portal: false`. `max_cards` must be an integer 1–5; the two
   flags must be booleans; unknown `user` keys are errors (`:194–196`).

Two emission decisions the form makes on top of the contract:

- **Zero and blank entries are omitted.** The optimizer reads absent categories
  as `spend.get(cat, 0)`, so omission is lossless, and the emitted file stays
  as short as the example file. A `merchant_spend` block with no nonzero
  carve-outs is omitted entirely — legal because `parse_profile` does
  `raw.get("merchant_spend") or {}` (`:173`).
- **All six `user` keys are always emitted**, defaults included. The file is
  the user's persistence (§1), so it should be self-documenting and editable
  without consulting docs — the same choice the example file makes.

## 3. Form structure and UX

The page reuses the two-column shell of `tools/card-entry-form.html` verbatim:
form blocks in the left column; a sticky right rail with a totals card, a
checks card (errors/warnings, §4), and a live YAML preview with **Copy** and
**Download** buttons. Same CSS-variable theming: light/dark from
`prefers-color-scheme` plus a manual toggle stamping `data-theme` on `:root`.

### 3.1 Category rows

All 13 non-pseudo categories are always visible, in `categories.yaml` file
order, with labels taken directly from the registry's `label` field
(e.g. `groceries` → "Groceries / supermarkets", `other` → "Everything else").
No progressive disclosure, no "add a category" picker: the fixed list is a
recall checklist — seeing "Drugstores & pharmacies" prompts spend the user
would forget in a free-form UI. Each row is the label, a single dollar input,
and a grey conversion annotation (§3.2). Blank and `0` are equivalent
(both omitted on emission, §2).

### 3.2 Units: monthly/annual toggle

A single global toggle — **per month | per year** — applies to all amount
inputs at once, defaulting to *per month* (how people actually estimate
groceries and streaming). Rules:

- Canonical state is **integer annual cents** per category/merchant, regardless
  of display mode. Monthly entry: `round(input × 100) × 12`; annual entry:
  `round(input × 100)`.
- Each nonzero row shows a live grey annotation in the *other* unit
  ("≈ $8,000/yr" under a $666.67/mo entry, "≈ $667/mo" under $8,000/yr),
  so the toggle never surprises.
- Toggling converts what each input *displays* but never mutates canonical
  state — toggling back and forth any number of times is a no-op.
- Emission divides cents by 100, printing an integer when whole and two
  decimals otherwise (`8000`, not `8000.0`; `641.67`, not `641.6700000001`).

Integer-cents math is what makes the carve-out inequality (§4 E3) exact —
comparisons happen on integers, safely inside the optimizer's `EPS = 1e-9`
tolerance, so the form can never emit a file the optimizer rejects on a
floating-point hair.

### 3.3 Merchant carve-outs

Merchants nest **under their parent category row**, behind a per-category
"break out specific merchants" disclosure — closed by default so the common
case (no carve-outs) shows a 13-row form. The mapping comes from
`merchants.yaml[m].category`:

| parent category   | merchants                                    |
|-------------------|----------------------------------------------|
| `groceries`       | `whole_foods`, `walmart`, `target`, `costco` |
| `transit`         | `uber`, `lyft`                               |
| `dining`          | `dunkin`                                     |
| `online_shopping` | `amazon`                                     |

Categories with no merchants show no disclosure. An open disclosure shows a
live budget line — "carve-outs $3,000 of $8,000 groceries" — turning the
contract's "counted INSIDE the parent, not in addition" into something
visually literal rather than a rule to remember.

### 3.4 "About you" block

Maps 1:1 to the `user:` mapping:

- `credit_tier`: required `<select>` with a blank "— select —" placeholder and
  the five `TIER_ORDER` values. Deliberately **no default** — this is the one
  field the form cannot guess, and it gates the optimizer's approval filter.
- The five optional fields, pre-filled to `USER_DEFAULTS` so an untouched block
  emits exactly the defaults: `valuation_mode` and `optimize_for` as two-option
  selects, the two flags as checkboxes, and `max_cards` as a 1–5 `<select>` so
  an out-of-range value is impossible by construction.
- Each field gets a one-line "why" hint (e.g. *floor = conservative cash-value
  of points; optimistic = transfer-partner value*). Hints for `credit_tier` may
  cite generic FICO bands (e.g. "very good ≈ 740–799") — that is UI guidance,
  not repo data, and must be commented as such in the source, since the repo's
  tiers are ordinal keys, not score ranges.

### 3.5 Totals card

The rail's totals card shows, always in annual terms regardless of the unit
toggle: total annual spend (sum of category amounts — carve-outs are
**excluded**, they're already inside their parents), a secondary "≈ $X/mo"
line, and a nonzero-category count ("7 of 13 categories"). When `other` is 0
but at least one other category is nonzero, show the single non-blocking nudge
(§4 W1): everything-else spend is where flat-rate cards earn, so leaving it
$0 usually understates the portfolio.

## 4. Client-side validation

Standing constraint: the form's validation is an **exact mirror of
`parse_profile` — never a superset**. The form must not block any profile the
optimizer would accept; a stricter form silently narrows the product. Every
check below cites the `optimize.py` line it mirrors.

Guaranteed **by construction** (no runtime check needed): top-level key set
and `user` key set (the form emits a fixed shape); category and merchant keys
registry-only (rows come from embedded registry copies, §6); pseudo-categories
excluded (not embedded at all); enum values and `max_cards` range (selects).

Runtime errors — Copy and Download are disabled while any exists:

- **E1** — at least one category > 0 (mirrors `'spend' must be a non-empty
  mapping`, `:162–164`, since the form omits zero rows).
- **E2** — every amount is a finite number ≥ 0 (`_require_number`, `:148–150`);
  the browser's `type="number"` input plus cents parsing makes this mostly
  unreachable, but `-`, `e`-notation edge cases and pasted garbage still land here.
- **E3** — per parent category, carve-out sum ≤ category amount, compared in
  integer cents (`:184–189`). The message must cover the common trip-wire:
  a merchant amount entered while its parent category is $0/blank
  ("Costco $3,000 exceeds groceries $0 — enter the full groceries total first;
  carve-outs are counted inside it").
- **E4** — `credit_tier` selected (`:197–198`).

Exactly one warning class, **W1** (the `other == 0` nudge, §3.5). Warnings
never block Copy/Download.

## 5. YAML emission rules

The emitter is hand-rolled string building — no YAML library and no CDN
(blocked by `file://` and by the no-build constraint). This is safe because
every emitted value is a number, a boolean, or an `[a-z_]+` registry slug:
nothing ever needs quoting or escaping.

Fixed layout, matching the §2 sample byte-for-byte in structure:

1. The fixed 3-line header comment (generator name, run command, rules
   pointer), then a blank line.
2. `spend:` with its trailing comment, then one `  key: value` line per
   nonzero category **in registry order** (not entry order, not sorted —
   the order of `categories.yaml`).
3. Blank line, then `merchant_spend:` with its two-line trailing comment,
   nonzero merchants in registry order — the whole block omitted when no
   merchant is nonzero (§2).
4. Blank line, then `user:` with all six keys in the fixed order shown in §2,
   with the same trailing comments as `examples/spend-profile.example.yaml`.

2-space indent throughout; numbers formatted per §3.2. Determinism, stated the
way `02-optimizer.md` states it: identical form state produces byte-identical
YAML — no timestamps, no randomness, no locale-dependent formatting.

## 6. Registry embedding and drift prevention

Because `file://` forbids fetching `data/meta/*.yaml` at runtime, the form
embeds JS copies of what it needs — the `tools/card-entry-form.html` pattern,
which `categories.yaml`'s header comment already anticipates ("shared by card
data … and the manual spend-entry UI"). Embedded constants:

- `SPEND_CATS`: 13 `{key, label}` entries, registry order, **pseudo-categories
  not included** — the "no pseudo-categories in profiles" rule becomes
  structural rather than checked.
- `MERCHANTS`: 8 `{key, label, category}` entries.
- `TIERS`: the 5-element `TIER_ORDER` list.
- `USER_DEFAULTS`: the five optional-key defaults.

`TIERS` and `USER_DEFAULTS` mirror **`scripts/optimize.py`**, not a
`data/meta/` registry — so the build session's CLAUDE.md sync rule must name
the script explicitly, extending the existing embedded-registry bullet: *when
`categories.yaml`, `merchants.yaml`, or `optimize.py`'s `TIER_ORDER` /
`USER_DEFAULTS` change, update `tools/spend-profile-form.html`'s embedded
copies in the same change.*

Rejected alternatives: runtime `fetch()` of the registries (blocked under
`file://`); build-time generation of the constants (there is no build step,
and creating one is exactly what step 4 defers).

## 7. Output affordances and workflow

### 7.1 Copy and Download

**Copy YAML** (clipboard) and **Download** (Blob + object URL, like
`card-entry-form.html`). Download always uses the fixed filename
`spend-profile.yaml`, so the command printed in the file header and in the
form's lede is copy-pasteable as-is.

### 7.2 Workflow guidance in the form

The form's lede echoes the §1 usage loop and adds one warning: the profile is
personal financial data — keep it out of the repo (save it anywhere outside
the working tree, or add it to a local ignore). The repo's committed profile
surface is `examples/spend-profile.example.yaml` only.

### 7.3 Deferred: YAML import / re-edit

V1 is emit-only. Justification: the emitted file is ~15 lines and trivially
hand-editable; keeping the browser tab open covers same-session iteration; an
importer doubles the validation and drift surface (every parse rule mirrored a
second time, in reverse); and `card-entry-form.html` set the emit-only
precedent. Named future enhancement, if re-editing proves painful: a strict
~30-line line-based importer that accepts only lines matching the §5 emitter's
own output shape and refuses the whole file on any unrecognized line — never a
general YAML parser.

## 8. Implementation plan (for the build session)

One file: `tools/spend-profile-form.html`. No JS test harness (none exists for
`card-entry-form.html` either); acceptance is manual but exact:

1. **Golden round-trip:** enter the numbers from
   `examples/spend-profile.example.yaml` (monthly mode: enter the monthly
   equivalents; annual mode: the file's numbers). The downloaded YAML must be
   accepted by `parse_profile`, and `python3 scripts/optimize.py --profile`
   output against it must be identical to running against the example file.
2. **Errors:** exercise E1–E4 each (all-blank form; a negative/garbage amount;
   Costco carve-out exceeding groceries, including the parent-is-blank case;
   no tier selected) and confirm Copy/Download disable and re-enable.
3. **Units:** toggle monthly↔annual repeatedly with mixed entries; canonical
   state (and the YAML preview) must not change.
4. **Omission:** a profile with no carve-outs must emit no `merchant_spend:`
   block; zero/blank categories must not appear.

Ancillary changes required **at build time, not now**:

- Extend CLAUDE.md's embedded-registry bullet to cover the new form, including
  the `optimize.py` constant mirrors (§6).
- Extend the `docs/architecture.md` diagram with the new form (per the repo's
  diagram-maintenance rule — the diagram gains the component only when it
  ships).
- Document the form wherever `tools/card-entry-form.html` is indexed
  (README/docs), with the §1 usage loop.
